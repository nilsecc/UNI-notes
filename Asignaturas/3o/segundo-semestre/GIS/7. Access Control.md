- Close Policy: si no dice nada se deniega el acceso

# DAC, MAC, RBAC
- DAC Models: (Discretionary) los propios usarios son los que gestionan los permisos de los objetos
- MAC Model: (Mandatory), no los usarios sino un admin, Se usa mas. Usa multilevel security Policy, Lo que define el sistema son niveles de seguridad para usarios y objetos, cada objeto y subject se le asigna un nivel de seguridad: Top secret, Secret, Confidential, Unclassified
	- Si estas en Secret Puedes leer en ese y de abajo y en w puede escribir en Secret hacia arriba pero no hacia abajo para no filtrar a niveles inferiores
	- Sistemas estrictos 
	- La asignacion de niveles de subject no es trivial: flow NSA trabajas en docs de poca importancia y te aumentan te suben a Secret
	- Pueden tener varios category set para cada cosa {Army, Navy}
- RBAC (): Los permisos se asocian a roles y los usarios a roles, un rol no es lo mismo que un grupo 
	- Usan el estandard del NIST
		- Core RBAC: Users <--> Role <--> Permissions. Cuando un usario entra se le inicia session y se le activan unos roles y no todos (MINIMO PRIVILEGIO)
		- Hierarchical RBAC: Puedes definir jerarquias entre roles
		- Constraint RBAC: Separacion de tareas
			- SSD: Se aplica a la asignacion, un uiser nunca puede pertenecer a los dos
			- DSD : En la sesion, puede pertenecer a los dos pero al entrar entra como uno
			- SD  un conjunto de roles que son mutuamente excluyentes tipo un usuario no puede estar en R1 y R2 a la vez. significa que para hacer algo critico necesito dos permisos entonces si estan separados necesito dos usuarios separados o 3 o 4
