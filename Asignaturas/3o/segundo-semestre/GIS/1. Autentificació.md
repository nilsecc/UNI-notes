## User Authentification

Identificar un usuari
- factors o categories: 
	- What you are: Dades biometriques, la forma de teclejar...
	- What you have: Clau privada, targeta del banc, dni, una clau fisica...
	- What You know: Password, aniversari...
	- Others: Donde estas (localizacion/IP), en que hora...

2 Factor Authentification: Se usan dos factores por ejemplo password y push notification

### Passwords

1. Enrollment: Guardar donde sea el nombre de usuario y el password (ej: base de datos guardamos user1 --> pass1, ...) Guardar el hash H*(p) por si se filtra la bd (el hash es unidireccional)
2. Verificación (login): El usuario demuestra que es el usuario1 usando el pass1 y el sistema consulta la bd y acepta/rechaza

**Ataques**
- Keylogger
- Shoulder surfing
- Ingenieria social
- Password recovery

**Categorias ataques password**
- Password Guessing
	- Online: limitar numero de intentos de acceso (rate-limit Attempts), el tiempo de espera, usar Captha --> Esto puede hacer que el atacante bloquee el acceso al usuario haciendole que se le active el rate-limit por ejemplo.
	- Offline: Conseguir la BD de passwords e intenta adivinar el password que hay detras del hash (preimagenes y colisiones, probar valores y ver si da el mismo hash (***fuerza bruta***) + usar ***diccionarios***)
		- Fuerza bruta
		- Dictionary
			- En claro (clear text) Que la funcion hash realmente no sea una funcion hash (los hashes son super rapidos) queremos que tenga un coste computacional rapido 
			- Precomputed (password + hash) Usar sal para complicar al atacante
- Defenses:
	- Password Streching: Hacer que la funcion hash tenga un coste computacional grande H*() y que sean resistentes hasta a GPU (anti mineria pcs)
	- Salt: 
		- Enrollment: u1,p1
		- Random string: s1
		- BD -> {u1: s1,H*(p1,s1)}
		 - Previene ataque de precomputo porque ahora no es passwors y hash  es el hash del passowrd cn la sal
	- Salt and pepper: Ofuscas la sal
